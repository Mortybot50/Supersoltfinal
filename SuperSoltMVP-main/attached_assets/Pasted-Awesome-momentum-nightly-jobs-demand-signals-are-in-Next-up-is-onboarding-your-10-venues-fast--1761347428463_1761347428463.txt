B) Server parsers & mappers

Create: lib/imports/parsers.ts
	•	Server-side CSV parser (no client parsing). Use papaparse or csv-parse/sync.
	•	Export:

export type Parsed = { headers: string[]; rows: any[] }; // rows are objects keyed by headers
export async function parseCsv(text:string): Promise<Parsed> { /* detect delimiter, handle BOM, trim headers */ }



Create: lib/imports/mappers.ts
	•	Export a mapper per type that takes { rows, mapping } and returns normalized rows (with units/cents coerced).
	•	Common helpers:
	•	toCents(v) accepts number/string like 12.50 → 1250.
	•	parseUnit(v) maps kg/g/l/ml/each.
	•	dateISO(v, tz='Australia/Melbourne') → YYYY-MM-DD (zeroed local).
	•	Mappers (MVP fields):
	•	ingredients: name, purchase_unit, preferred_supplier, supplier_sku, pack_size, pack_unit, pack_cost(¢|$), cost_per_unit_cents(optional)
	•	menu_items: name, price(¢|$), tax(optional)
	•	recipes: menu_item_name|id, ingredient_name|id, qty, unit, yield_pct(optional), wastage_pct(optional)
	•	staff: name, email, role, hourly_rate(¢|$)
	•	sales: date, menu_item_name|id, qty, unit_price(¢|$ optional) -> derive revenue_cents
	•	stock: ingredient_name|id, qty, unit (on-hand snapshot)

Return { ok: true, normalizedRows: [...] } or row-level { status:'error', message }.

Create: lib/imports/committers.ts
	•	Export one committer per type, each performing idempotent upserts scoped to venue (use name or external_id if provided). Wrap in a transaction.
	•	ingredients:
	•	Upsert ingredients on (venue_id, name).
	•	If supplier fields given, upsert ingredient_suppliers (preferred) and compute cost_per_unit_cents = pack_cost_cents / pack_size_in_base_units.
	•	menu_items:
	•	Upsert menu_items (venue_id, name) with price_cents.
	•	recipes:
	•	Resolve menu_item_id and ingredient_id by name if ids not provided.
	•	Upsert/remove recipe_lines so the recipe exactly matches the import slice for that menu_item_id.
	•	staff:
	•	Upsert users by email (create if missing), upsert memberships for venue/org, store hourly_rate_cents in staff table if you have one.
	•	sales:
	•	Upsert or insert daily_sales rows (aggregate by date + menu_item_id). If unit_price provided, set revenue_cents = qty * unit_price; else compute from menu price.
	•	stock:
	•	Insert stock_movements as a snapshot movement type (e.g., stock_take) to set on-hand to the given qty (compute delta if you already track balance).

Return { created, updated, skipped, errors } and update import_jobs/import_job_rows.

C) Import APIs (Node runtime)

1) Create job + upload CSV (raw text)
app/api/imports/create/route.ts (POST)
	•	Auth with requireOrg({ role: "manager" }).
	•	Body: { type: "...", filename: "...", text: "CSV_CONTENTS" }.
	•	Create import_jobs with status:'uploaded'.
	•	Parse server-side → store first 200 rows into import_job_rows with status:'pending' and the raw JSON (do not store whole CSV; you can keep text in memory only).
	•	Return { jobId, headers, sample: first 10 rows }.

2) Save column mapping
app/api/imports/[id]/mapping/route.ts (POST)
	•	Body: { mapping: Record<string,string> } (header → canonical field).
	•	Validate required fields for the type, set status:'validating'.
	•	Run mapper on all staged rows (for MVP, run on the rows you staged; if you prefer full file parse, re-parse from the original text passed in step 1—store it temporarily in memory or encode it into job_rows in chunks).
	•	For each row, write back status:'ok'|'error' + message.
	•	If any errors, set job status:'ready' only when zero errors; otherwise keep status:'validating' and return the errors summary.
	•	Return { ok:true, errorsCount, preview:[...] }.

3) Commit import
app/api/imports/[id]/commit/route.ts (POST)
	•	Checks job status:'ready'.
	•	Calls the relevant committer in a transaction.
	•	Updates job to status:'importing' while running, then done or failed with finished_at.
	•	Return per-type counts { created, updated, skipped, errors }.

4) Job status
app/api/imports/[id]/route.ts (GET)
	•	Returns job, errors count, and a small sample of error rows.

5) Templates
app/api/imports/templates/route.ts (GET)
	•	Query ?type=ingredients|menu_items|recipes|staff|sales|stock
	•	Return a CSV string with header row and 3 example rows (use safe demo values).

E) Validation rules (per type, enforced server-side)
	•	ingredients: name (req), purchase_unit (one of g/kg/ml/l/each), if pack_cost is in dollars, convert to cents; compute cost_per_unit_cents if not provided.
	•	menu_items: name (req), price in cents or dollars; tax optional.
	•	recipes: must resolve menu item + ingredient; qty > 0; unit compatible with ingredient base; yield/wastage default to 100%/0%.
	•	staff: unique email; role ∈ {FOH, BOH, Bar, Manager} (match your roles); hourly_rate required (convert).
	•	sales: date valid; menu item resolvable; qty ≥ 1; if unit_price missing, pull menu price; insert aggregated by day+item.
	•	stock: ingredient resolvable; unit compatible; update via stock movement snapshot (do not directly set balance).

Always return 200 with structured row errors; never 400 for “bad data”.

F) Idempotency, safety, and audit
	•	Scope all upserts by (venue_id, …).
	•	Where you rely on names, trim().toLowerCase() for matching; keep original casing in stored names.

	•	Wrap commit in a single transaction per type to avoid half-imports.
