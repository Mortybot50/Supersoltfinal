🔧 PROMPT 7 / 20 — Purchases & Receiving (POs from Order Guide → Receive → On-hand & Cost)

Goal
Enable managers to create Purchase Orders from the (edited) Order Guide, send them (status change), and receive full/partial deliveries that update on-hand and average cost. Keep everything multi-tenant and idempotent. Do not remove existing endpoints.

⸻

0) Schema (new tables + safe enums)

Add three tables: purchase_orders, purchase_order_lines, stock_movements. If you already have similar tables, adapt names/columns.

// db/schema.ts
export const purchase_orders = pgTable("purchase_orders", {
  id: uuid("id").defaultRandom().primaryKey(),
  org_id: uuid("org_id").notNull(),
  venue_id: uuid("venue_id").notNull(),
  supplier_id: uuid("supplier_id").notNull(),
  number: varchar("number", { length: 32 }).notNull(),  // e.g. PO-2025-0012
  status: varchar("status", { length: 16 }).notNull().default("DRAFT"), // DRAFT|SENT|PARTIAL|RECEIVED|CANCELLED
  currency: varchar("currency", { length: 8 }).notNull().default("AUD"),
  expected_date: date("expected_date"),
  notes: varchar("notes", { length: 512 }),
  subtotal_cents: integer("subtotal_cents").notNull().default(0),
  tax_cents: integer("tax_cents").notNull().default(0),
  total_cents: integer("total_cents").notNull().default(0),
  created_by: uuid("created_by").notNull(),
  created_at: timestamp("created_at", { mode:"date"}).defaultNow().notNull(),
  sent_at: timestamp("sent_at", { mode:"date"}),
  received_at: timestamp("received_at", { mode:"date"}),
});
createIndex("idx_po_org_venue_status").on(purchase_orders, t => [t.org_id, t.venue_id, t.status]);

export const purchase_order_lines = pgTable("purchase_order_lines", {
  id: uuid("id").defaultRandom().primaryKey(),
  po_id: uuid("po_id").notNull(),
  ingredient_id: uuid("ingredient_id").notNull(),
  // store pack info at time of order so later changes don’t break history:
  pack_label: varchar("pack_label", { length: 64 }),       // e.g. "6 × 1kg" or "1kg"
  base_uom: varchar("base_uom", { length: 8 }).notNull(),  // "g" | "ml" | "each"
  base_qty_per_pack: integer("base_qty_per_pack").notNull(), // e.g. 1000 (g) or 1000 (ml) or 1
  packs_ordered: numeric("packs_ordered", { precision: 12, scale: 3 }).notNull(), // allow fractional, we’ll ceil on send
  packs_received: numeric("packs_received", { precision: 12, scale: 3 }).notNull().default("0"),
  pack_cost_cents: integer("pack_cost_cents").notNull(), // cost per pack at order time
  line_total_cents: integer("line_total_cents").notNull(), // packs_ordered * pack_cost
});
createIndex("idx_pol_po").on(purchase_order_lines, t => [t.po_id]);

export const stock_movements = pgTable("stock_movements", {
  id: uuid("id").defaultRandom().primaryKey(),
  org_id: uuid("org_id").notNull(),
  venue_id: uuid("venue_id").notNull(),
  ingredient_id: uuid("ingredient_id").notNull(),
  type: varchar("type", { length: 16 }).notNull(), // RECEIPT | ADJUSTMENT | WASTE
  qty_base: integer("qty_base").notNull(),         // in base units (g/ml/each)
  unit_cost_cents: integer("unit_cost_cents").notNull().default(0), // average per base unit for valuation
  ref_type: varchar("ref_type", { length: 16 }),   // "PO"
  ref_id: uuid("ref_id"),
  occurred_at: timestamp("occurred_at", { mode:"date"}).defaultNow().notNull(),
});
createIndex("idx_mov_org_venue_ing").on(stock_movements, t => [t.org_id, t.venue_id, t.ingredient_id, t.occurred_at]);

If your existing inventory uses a separate inventory_on_hand table or a column on ingredients, keep it—we will update on-hand in addition to writing movements. Always store movements in base units (“g”, “ml”, “each”).

2) Create POs from the Order Guide (using edited packs)

New API:
	•	POST /api/purchases/create-from-order-guide
	•	Body: { start, days, safetyDays, overrides: [{ ingredientId, supplierId, recommendedPacks, packCostCents?, packLabel?, baseUom, baseQtyPerPack }] }

	•	Each line captures pack metadata (label/uom/base per pack) and pack_cost_cents from supplier/ingredient config (or override). line_total_cents = ceil(recommendedPacks) * pack_cost_cents.
	•	Returns { pos: [{ id, number, supplierName, totals }] }.

4) Status transitions (API)
	•	POST /api/purchases/:id/send
	•	Requires DRAFT. Rounds each packs_ordered = ceilPacks(packs_ordered).
	•	Recalculate totals. status = SENT, sent_at = now().
	•	Idempotency: accept header Idempotency-Key (store hash on PO row; ignore duplicates).
	•	POST /api/purchases/:id/cancel
	•	Allowed only when DRAFT. status = CANCELLED.

⸻

5) Receiving goods (partial or full)
	•	POST /api/purchases/:id/receive
	•	Body: { lines: [{ lineId, packsReceived, packCostCents? }], occurredAt? }
	•	For each line:
	•	Increase packs_received += packsReceived.
	•	Compute receiptQtyBase = packsReceived * base_qty_per_pack.
	•	Determine per base unit cost = (packCostCents || line.pack_cost_cents) / base_qty_per_pack.
	•	Write a stock_movements row: { type:"RECEIPT", qty_base: receiptQtyBase, unit_cost_cents: perBaseCost, ref_type:"PO", ref_id:po_id }.
	•	Update on-hand for the ingredient (in base units).
	•	Update weighted average cost for the ingredient (if you store average cost on ingredient or inventory table).
	•	After processing all lines:
	•	If all packs_received >= packs_ordered → status = RECEIVED, received_at = now()
	•	Else → status = PARTIAL.

If you don’t yet store avg cost, just maintain on-hand for now; keep the movement log for future costing.

UI Receive modal
