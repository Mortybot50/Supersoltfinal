2) One dashboard data endpoint
- New route: app/api/dashboard/summary/route.ts  (runtime nodejs)
- Input: ?period=day|week|month&start=YYYY-MM-DD
- Output:
  {
    kpis: {
      salesCents: number,
      cogsPct: number | null,
      labourPct: number | null,
    },
    chart: {
      days: Array<{ date: string, actualCents: number, forecastCents: number }>
    },
    topProducts: Array<{ id: string, name: string, valueCents: number, pctOfSales: number, changePct: number }>,
    forecastByItem: Array<{ menuItemId: string, qty: number, revenueCents: number }>
  }
- Implementation details:
  - Use getWindow() for the range.
  - Actuals:
     * Sum by date for the window from your existing sales tables.
  - Forecast:
     * Ensure coverage: call POST /api/forecast/generate for the window (idempotent).
     * Compute per-day **forecast qty** from sales_forecasts.
     * Compute per-day **forecast revenue** = Σ(qty_item * current_price_cents_of_item).
  - KPIs:
     * salesCents = Σ actualCents in window (if future only, this can be 0).
     * cogsPct: theoretical COGS based on recipes * actual quantities in the window (fallback null if data incomplete).
     * labourPct: scheduled cost in window / salesCents (fallback based on forecast if salesCents==0).
  - TopProducts:
     * For the window, aggregate actual revenue by item.
     * pctOfSales = itemRevenue / totalActualRevenue (0 if denom 0).
     * changePct = ((thisWindowRevenue - prevWindowRevenue) / max(prevWindowRevenue, 1)) * 100
       where prevWindow is the same length immediately before start.
